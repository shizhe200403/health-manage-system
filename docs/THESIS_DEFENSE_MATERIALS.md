# 毕业设计论文与答辩材料

## 1. 课题名称

智能菜谱推荐与营养健康管理系统的设计与实现

## 2. 课题背景与研究意义

随着居民健康意识增强，饮食管理已经从“吃饱”转向“吃得科学、吃得长期可坚持”。但现实中多数用户在饮食管理上存在几个共性问题：

- 不清楚自己每天应该摄入多少热量和蛋白质。
- 缺少可执行的菜谱方案，知道原则但不会落地。
- 饮食记录成本高，导致计划难以长期坚持。
- 市面上的普通菜谱产品强调“做法展示”，对个性化健康约束支持不足。
- 社区内容较多，但缺少与用户健康目标联动的推荐与反馈闭环。

本课题以真实用户饮食管理场景为切入点，设计并实现一套集“菜谱推荐、营养分析、饮食记录、目标追踪、社区互动、报表输出、云端部署”于一体的产品化系统。系统不仅服务于毕业设计展示，也具备进一步扩展为商用健康饮食平台的工程基础。

## 3. 研究目标

本系统的总体目标是构建一个面向普通用户的智能饮食健康管理平台，具体目标如下：

- 实现用户从注册登录、资料建档到饮食管理的完整业务闭环。
- 基于用户资料、健康约束和行为偏好生成可解释的菜谱推荐结果。
- 支持饮食记录、营养统计和目标进度跟踪，帮助用户长期坚持。
- 提供社区互动和内容治理机制，增强用户留存与内容安全性。
- 支持周报、月报和 PDF 导出，满足阶段性复盘和成果展示需求。
- 按照云服务器上线标准设计工程结构、部署方式和运维文档。

## 4. 需求分析

### 4.1 目标用户

- 希望减脂、增肌或维持体重的普通个人用户。
- 有轻度慢病管理需求的人群，如控糖、低脂、少盐饮食人群。
- 需要通过阶段性数据输出进行饮食干预记录的学生或家庭用户。
- 希望沉淀健康饮食内容、交流经验的社区用户。

### 4.2 核心业务场景

1. 新用户首次注册后，填写身高、体重、目标体重、活动水平、饮食偏好和健康限制，系统自动计算 BMI、热量目标和蛋白目标。
2. 用户浏览首页推荐菜谱，查看“为什么推荐给我”，判断是否符合自身饮食目标。
3. 用户快速录入某一餐的饮食情况，系统累计营养数据并形成周/月趋势。
4. 用户设置减脂或控糖目标，持续记录进展并查看完成度。
5. 用户在社区发布饮食经验、评论他人帖子，并通过举报机制控制不良内容。
6. 用户在一个周期结束后导出周报或月报，用于自我复盘、健康管理或成果展示。

### 4.3 功能需求

- 用户认证与档案管理
- 菜谱管理与收藏
- 营养分析与指标计算
- 饮食记录与趋势统计
- 健康目标与进度追踪
- 推荐结果生成与推荐理由解释
- 社区发帖、评论、举报与治理
- 外部营养数据库接入与降级容错
- 报表生成与 PDF 导出

### 4.4 非功能需求

- 可维护性：前后端分离，模块边界清晰。
- 可扩展性：外部 API、推荐算法、消息通知可继续扩展。
- 安全性：基于 JWT 认证、接口权限控制、敏感操作鉴权。
- 可靠性：外部接口异常时可降级返回，避免核心流程中断。
- 可部署性：支持 Docker Compose 部署到云服务器。

## 5. 系统总体设计

### 5.1 架构设计

系统采用前后端分离架构，主要由以下几层组成：

- 前端表现层：基于 Vue 3 + Vite + TypeScript + Element Plus，负责页面展示、交互和状态管理。
- 后端接口层：基于 Django REST Framework，提供统一 RESTful API。
- 业务服务层：封装用户档案、菜谱、营养、记录、推荐、社区、报表等业务逻辑。
- 数据存储层：PostgreSQL 存储核心业务数据，Redis 用于缓存与异步任务协同。
- 任务处理层：Celery 用于报表等高成本任务扩展。
- 外部服务层：接入 USDA、Nutritionix、Edamam、OpenFoodFacts 等开放平台，为营养数据补全提供能力。

### 5.2 技术选型与原因

#### 后端

- Django 5.2：成熟稳定，适合快速构建具备管理后台、权限和 ORM 能力的业务系统。
- Django REST Framework：接口开发效率高，认证、序列化、权限体系完备。
- SimpleJWT：适合前后端分离场景，便于移动端和 Web 统一鉴权。
- PostgreSQL：适合关系复杂、数据一致性要求高的业务系统。
- Redis：适合作为缓存和 Celery 中间件。
- Celery：适合异步报表、定时任务、后续通知推送等任务扩展。

#### 前端

- Vue 3：语法简洁，适合中小型团队快速实现复杂后台类系统。
- Vite：启动快、构建效率高，适合毕业设计和产品化原型的持续迭代。
- TypeScript：增强接口约束和可维护性。
- Pinia：作为轻量级状态管理方案，适合认证状态和用户信息管理。
- Element Plus：能快速形成可用且成熟的中后台交互界面。

#### 部署

- Docker Compose：适合单机云服务器部署，部署简单，组件边界清晰。
- Nginx：统一托管前端静态资源并代理后端接口。
- Gunicorn：用于 Django 服务进程管理。

### 5.3 商用化导向的设计考虑

- 数据结构为长期迭代预留来源字段、审计字段和状态字段。
- 外部开放平台接入设计为代理层，避免把第三方协议直接暴露到前端。
- 推荐结果保留原因文本，增强可解释性，降低用户对推荐黑盒的不信任。
- 社区模块与核心健康业务解耦，既能拉动活跃，也便于后续单独治理。
- Docker 化部署和运维文档同步交付，降低上线门槛。

## 6. 详细功能设计

### 6.1 用户与档案模块

系统支持用户名、邮箱、手机号三种账号方式登录，降低用户记忆成本。用户首次进入系统后可一次性完善基础资料、健康档案和饮食约束。后端提供原子化的全档案更新接口，避免前端多次提交导致的数据不一致问题。

### 6.2 菜谱与食材模块

系统支持食材、菜谱、步骤、营养汇总等结构化数据管理。菜谱具备菜系、口味、难度、餐次、份量等属性，可用于后续检索、推荐和统计。菜谱支持收藏操作，并将行为反馈写入用户行为表，为推荐排序提供依据。

### 6.3 营养分析模块

系统根据用户身高、体重、活动水平等信息计算 BMI、热量目标和蛋白目标，并通过菜谱营养汇总与饮食记录形成摄入分析结果。该模块为推荐、记录统计和报表输出提供统一指标基础。

### 6.4 饮食记录模块

用户可以按日期、餐次快速录入饮食记录，支持记录菜谱条目和备注信息。系统对记录进行汇总，生成周、月维度的摄入统计与趋势数据，为后续目标评估和报表展示提供支撑。

### 6.5 健康目标模块

系统支持体重管理等目标的创建和进度追加。用户可以设置目标值、当前值、时间范围和说明，随后按时间录入进展数据。系统据此形成目标趋势和阶段性完成情况。

### 6.6 推荐模块

推荐模块采用“规则过滤 + 内容召回 + 行为排序”的轻量混合方案：

- 规则过滤：优先剔除与用户过敏、慢病约束、饮食偏好冲突的菜谱。
- 内容召回：基于餐次、饮食类型、菜谱标签等召回候选集合。
- 行为排序：结合收藏、浏览、记录等行为信息进行简单打分排序。
- 结果解释：对每条推荐生成可读的理由文本，提升系统可信度。

该方案相较纯机器学习方法更适合当前毕业设计阶段，优点是实现成本可控、上线后行为闭环清晰、结果可解释性更强。

### 6.7 社区与治理模块

社区模块包括帖子发布、评论互动和举报机制。为了降低内容风险，帖子与评论设计了状态字段和审核字段，后端也预留了内容治理入口。该设计体现了系统从“功能可用”走向“上线可管”的商用化思路。

### 6.8 报表模块

系统支持周报、月报和 PDF 文件导出。报表内容聚合饮食记录、目标进度、营养指标和阶段性总结，既可以作为用户自我复盘工具，也可以作为毕业设计成果展示的一部分。

### 6.9 外部开放平台接入模块

系统设计了统一的外部 API 代理层，已接入以下开放平台的代理能力：

- USDA FoodData Central
- Nutritionix
- Edamam
- OpenFoodFacts

外部接口失败时，系统返回降级标识并保留统一响应结构，避免页面直接崩溃。该设计体现了面向真实生产环境的容错意识。

## 7. 数据库设计说明

系统数据库围绕用户、菜谱、营养、行为、记录、目标、社区、报表和外部数据展开。核心设计原则如下：

- 使用规范化表结构保证核心业务数据一致性。
- 在推荐、报表等场景保留日志型数据，支撑审计与分析。
- 对第三方营养数据保留来源和原始载荷，便于后续比对和清洗。
- 对社区内容和举报信息保留状态与处理信息，满足治理需要。

数据库的核心表包括：

- 用户相关：`user`、`user_profile`、`user_health_condition`
- 菜谱相关：`recipe`、`recipe_step`、`ingredient`、`recipe_ingredient`、`recipe_nutrition_summary`
- 行为推荐相关：`user_behavior`、`recommendation_log`、`user_feedback`
- 健康管理相关：`meal_record`、`meal_record_item`、`health_goal`、`health_goal_progress`
- 社区相关：`post`、`post_comment`、`content_report`

数据库结构参考文档见 [database_design.md](../database_design.md)。实际表结构以 Django migrations 为准。

## 8. 核心算法与方法说明

### 8.1 推荐方法

本系统当前推荐方法不是复杂的深度学习模型，而是适合中小规模产品落地的工程化混合推荐方法。其核心思想是：

1. 先过滤不适合用户的内容，保证推荐安全性和可用性。
2. 再从餐次、偏好、标签等维度召回可能合适的内容。
3. 最后用用户历史行为对结果重新排序。

这种方法的优势在于：

- 可解释性好，适合健康饮食场景。
- 冷启动成本低，新用户也能得到推荐。
- 工程实现简单，适合毕业设计与产品原型阶段。
- 后续可以平滑升级为向量召回、协同过滤或机器学习排序。

### 8.2 营养分析方法

营养分析基于用户身体指标和活动水平，综合计算：

- BMI 指数
- 基础代谢与热量目标
- 蛋白质摄入目标

系统在当前阶段重点关注可执行性最强的几个指标，而不是一次性引入过多复杂营养学参数。这样既降低用户理解门槛，也便于在前端页面中直观展示。

### 8.3 外部数据容错方法

由于第三方 API 的稳定性、速率限制和数据格式可能不同，系统在代理层采用以下方法：

- 统一出参结构，减少前端适配复杂度。
- 捕获异常并返回 `degraded` 标识。
- 为后续引入缓存、重试和熔断机制预留扩展空间。

## 11. 系统创新点与特色

### 11.1 面向真实场景的健康饮食闭环

系统并非单一菜谱展示工具，而是把“建档、推荐、记录、分析、目标、复盘”连接成一条闭环链路，更符合真实用户使用过程。

### 11.2 可解释的推荐机制

推荐不仅给出结果，还提供推荐原因文本，提升用户对系统推荐结果的理解和接受度。

### 11.3 外部开放平台能力接入

系统不是封闭数据库，而是具备与外部营养平台协同的能力，便于后续丰富食材库、条码库和营养分析精度。

### 11.4 商用化工程规范意识

本课题同步交付了 API 文档、架构文档、测试文档、编码规范、部署手册和生产运行手册，体现出产品化和工程化建设能力，而不仅是页面原型。

## 12. 存在问题与后续优化方向

虽然系统已具备较完整的业务框架，但距离成熟商用系统仍有进一步优化空间：

- 推荐算法仍以规则和轻量排序为主，尚未引入更强的数据驱动模型。
- 短信验证码登录、隐私设置、找回密码等账号能力仍属于规划项。
- 前端界面已覆盖主要页面，但缺少更丰富的空状态、错误态和细粒度交互反馈。
- 报表生成当前仍偏同步执行，后续应全面切换到异步任务链路。
- 运维监控、日志聚合、告警通知、备份恢复演练尚需上线前补齐。

## 14. 常见答辩问题与参考回答

### 问题 1：为什么选择 Django，而不是 Spring Boot 或 FastAPI？

参考回答：本课题强调的是完整业务系统快速落地。Django 在 ORM、后台管理、权限、模型组织和生态稳定性方面非常成熟，适合毕业设计阶段构建中等复杂度业务系统。相比之下，FastAPI 更轻量，但在后台管理、复杂业务组织方面需要自行补更多工程能力。Spring Boot 也很成熟，但当前项目以 Python 生态中更便于实现推荐、营养分析和数据处理的技术路线为主，因此选择 Django 更匹配课题目标。

### 问题 2：推荐算法为什么没有直接使用机器学习模型？

参考回答：健康饮食推荐比普通内容推荐更强调可解释性和约束安全性。当前阶段采用规则过滤加轻量排序的混合方案，可以更稳定地处理过敏、慢病和饮食偏好等强约束场景，同时对新用户冷启动更友好。后续在数据积累足够后，可以继续升级为协同过滤或学习排序模型。

### 问题 3：系统如何体现“智能”？

参考回答：智能体现在三个方面。第一，系统会根据用户健康档案自动计算 BMI、热量和蛋白目标。第二，系统会依据用户资料和行为偏好生成个性化菜谱推荐。第三，系统能够基于历史记录形成统计、趋势和报表输出，而不是只做静态信息展示。

### 问题 4：系统如何保证上线可行性？

参考回答：系统在工程上采用前后端分离和 Docker 化部署方式，并配套了部署文档、运行手册、编码规范和测试文档。数据库以 Django migrations 为准，Nginx、Gunicorn、PostgreSQL、Redis 的服务边界清晰，具备部署到云服务器进行演示和继续迭代的基础。

### 问题 5：你的系统和普通菜谱网站的区别是什么？

参考回答：普通菜谱网站核心是内容浏览，而本系统重点是个性化健康管理。系统围绕用户档案、营养目标、饮食记录、目标进度和推荐解释构建闭环，更强调用户持续使用和健康管理价值。

### 问题 6：如果要继续商用化，优先会做什么？

参考回答：优先会补齐账号安全能力、支付前后的会员体系、日志监控、异步任务体系、隐私设置和推荐算法升级。同时会接入短信、邮箱、对象存储和运营后台，逐步把当前系统从毕业设计原型推进到可运营产品。
